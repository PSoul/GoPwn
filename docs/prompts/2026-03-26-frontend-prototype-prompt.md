# 前端高保真原型 Prompt

以下 Prompt 用于发送给专业的前端原型图 / 前端设计类 LLM。  
本 Prompt 负责定义产品结构、页面信息架构、业务流程、模块边界、关键交互和数据展示重点。  
视觉风格、配色、品牌语义、排版体系、动效语言、参考网站风格映射等内容，预留给后续单独补充。

---

你是一名资深 B 端产品设计师兼前端原型设计师，请基于以下产品需求，为一个“基于 LLM 编排内核的授权外网安全评估平台”设计一套完整的前端高保真原型。

## 一、你的任务边界

你需要完成的是：

1. 基于产品需求设计完整的信息架构与页面结构。
2. 设计后台管理系统的核心页面、页面关系、关键操作流程和页面状态。
3. 设计可直接进入高保真原型阶段的页面方案。
4. 在不改变产品核心逻辑的前提下，自主完成页面布局、组件组织、交互路径和内容呈现方式。

你必须遵守以下要求：

1. 这是一个后台管理系统，不是聊天产品。
2. 不要设计常驻聊天入口，不要把 LLM 作为前台对话主角。
3. 用户主要通过项目、资产、任务、审批、证据、系统设置等结构化界面操作平台。
4. LLM 固定在后端，作为编排内核与决策大脑，不作为首页的主交互入口。
5. 你的设计重点是“流程控制、项目管理、资产沉淀、审批与证据可视化”。
6. 不要改变下面定义的产品范围和工作流程。

## 二、视觉与样式输入

以下内容将由我后续单独补充，你需要严格吸收并用于高保真设计：

1. `UI/UX 能力技能或风格技能`
2. `参考网站`
3. `视觉方向`
4. `品牌或语义要求`
5. `动效与交互质感要求`

你在收到这些额外输入后，需要：

1. 保留本 Prompt 定义的产品结构与业务流程不变。
2. 将视觉层高保真化。
3. 在不破坏后台管理系统可用性的前提下，提升专业感、秩序感、信息密度控制和操作效率。

## 三、产品背景

这是一个面向单人研究员的授权外网安全评估工作台。

平台的目标是：当研究员给定一个目标，例如公司名、域名、URL、IP 或 CIDR 后，平台能够围绕常见的渗透测试流程持续推进，完成以下事情：

1. 信息收集
2. 资产发现
3. 资产关联与范围判定
4. 端口识别与服务识别
5. 技术指纹识别
6. 待验证项生成
7. 高风险动作逐项审批
8. 审批后的受控验证
9. 证据归档
10. 结果聚合
11. 报告输出与回归验证

平台的核心不是聊天，也不是展示大量工具按钮，而是：

- 项目流转
- 阶段推进
- 资产沉淀
- 任务调度
- 审批控制
- 证据管理
- 结果汇总

## 四、产品定位

请始终围绕以下定位进行设计：

1. 单人研究员工作台。
2. 后台管理界面。
3. 首版只覆盖匿名外网面，不做登录后测试。
4. 首版是单机一体化，但后续可扩展到分布式后端。
5. 高风险动作必须逐项人工审批。
6. 平台所有与目标交互的外部动作都通过统一的 MCP 能力层完成。

## 五、产品核心原则

1. 流程优先，不以聊天为中心。
2. 信息收集贯穿整个项目生命周期，而不是只在开头执行一次。
3. 渗透测试流程不是线性的，而是可以回流的。
4. 任何新发现的域名、子域、IP、端口、服务、页面入口、API 入口、管理后台、跳转目标、技术线索，都可能触发回流。
5. 平台必须让研究员随时知道：
   - 当前项目做到哪一步
   - 为什么停在这里
   - 下一步是什么
   - 有哪些待审批动作
   - 目前沉淀了哪些资产、指纹、证据和结果
6. 证据优先于结论，任何结论都应该可以追溯到任务、审批、执行和证据。
7. 平台必须体现 MCP 是一等公民，工具能力是可扩展接入的。

## 六、平台主流程

请基于以下“可回流阶段状态机”设计项目详情页和相关原型。

平台主阶段包括：

1. 授权与范围定义
2. 种子目标接收
3. 持续信息收集
4. 目标关联与范围判定
5. 发现与指纹识别
6. 待验证项生成
7. 审批前排队
8. 受控 PoC 验证
9. 证据归档与结果判定
10. 风险聚合与项目结论
11. 报告与回归验证

关键说明：

1. 信息收集是贯穿全程的常驻能力，不是一次性阶段。
2. 项目可能在后续阶段因为新线索而回流到前置阶段。
3. 项目需要有“主阶段”展示，但项目内部允许同时存在多个不同阶段的任务。
4. 用户在前台看到的是“项目整体主阶段 + 当前关键任务 + 沉淀信息面板”的组合。

## 七、整体信息架构

请设计一个标准后台管理系统的左侧导航，至少包含：

1. 仪表盘
2. 项目管理
3. 审批中心
4. 资产中心
5. 证据与结果
6. 系统设置

不要设计聊天入口，不要设计 AI 对话工作区。

## 八、页面设计要求

### 1. 仪表盘

仪表盘是全局概览页，进入系统后优先看到整体态势。

请至少体现以下信息：

- 项目总数
- 运行中项目数
- 已发现资产数量
- 已确认问题数量
- 待审批动作数量
- 最近新发现资产
- 最近执行任务
- 最近验证结果
- MCP 工具健康状态
- 当前异常或失败任务提示

请重点考虑：

- 信息分区方式
- 概览卡片与列表的关系
- 让研究员快速判断今天最需要处理什么
- 哪些信息适合放首页，哪些信息适合跳转到详情页

### 2. 项目管理

项目管理负责项目的增删改查和进入项目详情。

请至少设计：

- 项目列表页
- 新建项目页或弹窗
- 编辑项目页或弹窗
- 项目筛选与搜索
- 项目状态展示
- 当前阶段展示

项目基础字段至少包括：

- 项目名称
- 目标类型
- 目标种子
- 授权说明
- 范围规则
- 禁止动作
- 默认并发限制
- 默认速率限制
- 审批策略
- 创建时间
- 最近更新时间

### 3. 项目详情页

这是最核心的页面。

项目详情页必须以“阶段流转视角”为主轴，同时展示项目当前已沉淀的信息和结果。不要把项目详情页设计成一个普通 tab 列表页，而要突出“当前项目正在如何推进”。

页面必须包含：

1. 顶部摘要区
   至少包含：
   - 项目名称
   - 目标摘要
   - 当前主阶段
   - 整体状态
   - 待审批数量
   - 风险摘要
   - 最近更新时间

2. 阶段流转主区域
   需要清楚展示：
   - 阶段顺序
   - 当前所在阶段
   - 已完成阶段
   - 阻塞阶段
   - 回流提示
   - 下一步建议

3. 当前阶段待办区
   需要展示：
   - 当前关键任务
   - 当前阻塞原因
   - 哪些任务等待审批
   - 哪些任务等待依赖
   - 哪些任务执行失败需要处理

4. 项目沉淀信息区
   需要用标签页、分区或组合视图承载以下内容：
   - 已发现信息
   - 资产
   - IP / IP 段
   - 端口 / 服务
   - 指纹 / 技术栈
   - Web / API 入口
   - 漏洞 / 待复核问题
   - 审批记录
   - 证据与日志
   - 任务与调度

请重点设计：

- “阶段主视图”和“沉淀结果面板”如何共存
- 如何让研究员既能看整体推进，又能看具体结果
- 当发生回流时，页面如何表达“项目主阶段未重置，但有新线索回流到前置阶段”

### 4. 审批中心

审批粒度是“单个高风险动作审批”。

请至少设计：

- 审批列表页
- 审批详情页或侧滑面板
- 审批状态筛选
- 单条审批的阅读与决策操作

每条审批单至少展示：

- 目标对象
- 动作类型
- 风险等级
- 平台建议理由
- MCP 能力类别
- 具体工具
- 参数摘要
- 预期影响
- 执行前提
- 停止条件
- 审批状态
- 批准 / 拒绝 / 延后

请让研究员清楚看到：

- 为什么推荐执行这一步
- 批准之后会发生什么
- 不批准会对项目造成什么影响

### 5. 资产中心

资产中心需要体现“资产对象 + 当前识别画像 + 关联关系”。

资产对象包括但不限于：

- company
- domain
- subdomain
- ip
- cidr
- host
- port
- service
- website
- api
- page_entry

重要要求：

资产对象不仅要显示“是什么”，还要显示“当前识别成了什么”。例如：

- 22/tcp -> OpenSSH 某版本线索
- 3306/tcp -> MySQL 某版本线索
- 443/tcp -> nginx + 某 Web 技术栈线索

请至少设计：

- 资产列表页
- 资产详情页
- 资产关系展示方式
- 当前识别画像展示方式
- 与项目、任务、证据、漏洞的关联入口

### 6. 证据与结果

需要体现“证据优先于结论”。

请至少设计：

- 证据列表页
- 证据详情页
- Finding / 漏洞结果页
- 证据到结论的链路展示

需要支持展示：

- 原始输出
- 截图
- 结构化摘要
- 关联任务
- 关联审批单
- 关联资产
- 时间线
- 置信度
- 最终结论

### 7. 系统设置

系统设置需要体现平台的可控性和可扩展性。

请至少设计：

- MCP 工具管理
- MCP 能力分类管理
- 工具启用 / 禁用
- 工具版本与健康状态
- 默认并发设置
- 默认速率限制
- 默认超时
- 默认重试策略
- 审批策略
- 范围规则
- 紧急停止开关
- 审计与日志设置

请不要把系统设置做成普通的零散配置页，而要体现这是一个“可控的安全评估平台控制台”。

## 九、调度模块展示要求

调度模块必须在前台被感知到，不一定需要单独做一级导航，但必须在项目详情页或相关页面中有清晰体现。

调度模块至少要能让研究员看到：

1. 任务队列
2. 任务状态
3. 优先级
4. 依赖关系
5. 定时调度
6. 自动重试 / 人工重试
7. 并发与速率控制
8. 待审批阻塞
9. 手工暂停 / 恢复 / 终止

任务状态至少包括：

- pending
- ready
- waiting_dependency
- waiting_approval
- scheduled
- running
- succeeded
- failed
- needs_review
- cancelled

请在原型中让研究员明确知道：

- 为什么任务没有执行
- 是卡在依赖、审批、调度还是执行失败
- 哪些任务阻塞了项目推进

## 十、MCP 模块展示要求

请在原型中体现：

1. 平台所有外部目标交互动作都经过 MCP 能力层。
2. MCP 工具是可扩展接入的。
3. 每个工具具备能力类别、版本、风险级别、启用状态和默认限制。
4. 平台是按能力选工具，而不是写死某一个工具。
5. 高风险能力会触发审批门槛。

不要把 MCP 设计成“开发者内部概念不可见”，需要让研究员在系统设置、审批、任务或项目页中感知到它的存在。

## 十一、关键交互流程

请在原型中清楚表达以下关键流程：

### 1. 创建项目流程

从录入种子目标、填写授权和范围，到项目创建成功。

### 2. 项目推进流程

研究员进入项目后，可以查看：

- 当前主阶段
- 当前待办
- 当前阻塞
- 下一步建议
- 当前沉淀结果

### 3. 资产扩展与回流流程

当项目中发现新入口、新域名、新 IP、新服务时：

- 如何进入候选资产
- 如何进行目标关联与范围判定
- 如何触发回流到前置阶段
- 如何在前台表达这个回流过程

### 4. 审批流程

待验证项如何进入审批队列，研究员如何查看详情、批准、拒绝、延后。

### 5. 受控执行流程

审批通过后，任务如何进入调度并执行，如何生成证据并更新结果。

### 6. 证据查看与结果确认流程

研究员如何从执行记录进入证据，再从证据进入问题结论。

### 7. 异常与人工接管流程

任务失败、等待审批、等待依赖、超时、需要复核时，研究员如何接管。

## 十二、输出要求

请按以下结构输出你的设计结果：

### 1. 信息架构

输出系统整体站点地图、模块结构或页面树。

### 2. 页面清单

列出所有一级页面、二级页面、关键弹窗、抽屉、详情页和状态页。

### 3. 页面原型说明

逐页说明：

- 页面目标
- 页面区域划分
- 信息层级
- 主要模块
- 主要操作
- 页面之间的跳转关系

### 4. 关键流程原型说明

重点说明：

- 项目创建
- 项目推进
- 资产回流
- 单项审批
- 受控执行
- 证据查看
- 调度干预

### 5. 关键组件清单

至少包含：

- 阶段流转组件
- 项目摘要组件
- 审批卡片
- 任务队列表格
- 任务状态标识
- 资产画像面板
- 资产关系视图
- 证据详情面板
- MCP 工具配置表
- 系统控制面板

### 6. 示例数据建议

请为关键页面补充合理的中文示例数据，便于后续直接生成高保真原型。

## 十三、禁止事项

1. 不要把系统设计成聊天产品。
2. 不要把首页做成 AI 助手欢迎页。
3. 不要改变产品的匿名外网面边界。
4. 不要设计无人审批的高风险自动执行路径。
5. 不要把重点放在展示工具 Logo 或工具堆叠上。
6. 不要弱化项目流转、审批、证据和调度。
7. 不要把系统做成偏攻击平台的表达方式。

## 十四、最终目标

你的目标不是生成一套普通后台，而是生成一套真正适合“单人研究员进行授权外网安全评估”的高保真原型结构。

它必须同时满足：

1. 研究员能看懂全局
2. 研究员能管理项目
3. 研究员能追踪流程
4. 研究员能审批高风险动作
5. 研究员能查看资产与指纹画像
6. 研究员能查看证据与结果
7. 研究员能理解调度与阻塞
8. 研究员能在系统设置中管理 MCP 和平台控制策略

请基于以上要求，输出一套适合进入高保真原型设计阶段的完整方案。
