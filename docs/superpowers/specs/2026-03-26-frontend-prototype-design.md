# 基于 LLM 编排内核的授权外网安全评估平台前端原型设计文档

## 1. 文档目标

本文档用于定义“基于 LLM 编排内核的授权外网安全评估平台”首版前端高保真原型方案。该方案面向单人研究员，目标是在不改变既有产品边界的前提下，为后续前端原型开发提供稳定的页面结构、导航关系、关键交互和组件依据。

本次原型范围聚焦核心闭环页面：

1. `/login`
2. `/dashboard`
3. `/projects`
4. `/projects/new`
5. `/projects/[id]`
6. `/approvals`
7. `/assets` 与 `/assets/[id]`
8. `/evidence` 与 `/evidence/[id]`
9. `/settings`

设计主轴采用“流程指挥台型”方案，即以项目推进和阻塞清理为核心体验，同时兼顾审批、资产、证据与 MCP 控制面。

## 2. 设计基线与约束

### 2.1 产品约束

1. 平台是后台管理系统，不是聊天产品。
2. LLM 位于后端，承担编排与决策辅助角色，不作为首页或项目页主交互入口。
3. 用户主要通过项目、资产、任务、审批、证据、系统设置等结构化界面操作平台。
4. 高风险动作必须逐项人工审批，不允许存在无人审批的高风险自动执行路径。
5. 平台需要体现 MCP 是一等公民，且所有外部目标交互动作都经过 MCP 能力层。

### 2.2 视觉基线

1. 视觉气质继承现有后台模板：专业、克制、清晰、偏中性色。
2. 页面强调高信息密度和稳定扫描效率，而非营销化包装。
3. 风险、阻塞、审批、失败、回流等状态是主要视觉层级来源。
4. 登录页采用标准平台账号登录体验，使用账号、密码、验证码的后台入口表达。

### 2.3 设计原则

1. 流程优先，不以聊天为中心。
2. 证据优先于结论。
3. 主阶段不重置，但允许前置阶段因新线索回流。
4. 审批既属于项目，也必须支持跨项目集中处理。
5. 调度信息必须可见、可解释、可人工接管。

## 3. 信息架构

### 3.1 一级导航

1. 仪表盘
2. 项目管理
3. 审批中心
4. 资产中心
5. 证据与结果
6. 系统设置

### 3.2 路由树

```text
/login
/dashboard
/projects
/projects/new
/projects/[id]
/approvals
/approvals/[id]        # 可作为详情路由或抽屉深链
/assets
/assets/[id]
/evidence
/evidence/[id]
/settings
```

### 3.3 路由策略说明

1. 审批中心采用全局一级路由 `/approvals`，因为研究员需要集中清理跨项目阻塞。
2. 项目详情页内嵌项目级审批视图，确保在项目上下文中也能直接查看待审批动作与审批历史。
3. 首版不强制建设独立的 `/projects/[id]/approvals` 页面，但可在后续扩展为项目专属审批视图。
4. MCP 不单列为一级导航，而是在项目任务、审批详情、系统设置中持续露出，以保持可感知性。

## 6. 关键流程原型说明

### 6.1 创建项目流程

1. 研究员进入项目列表页并点击“新建项目”。
2. 录入项目名称、目标类型与目标种子。
3. 填写授权说明、范围规则、禁止动作。
4. 配置默认并发、默认速率和审批策略。
5. 确认创建后进入项目详情页，项目进入“授权与范围定义”或“种子目标接收”的初始推进状态。

### 6.2 项目推进流程

1. 研究员进入项目详情页。
2. 顶部摘要区先给出当前主阶段与整体状态。
3. 阶段流转主区域解释当前卡点和下一步建议。
4. 当前阶段待办区展示关键任务、等待审批、等待依赖和失败待处理项。
5. 沉淀信息工作台支持查看资产、入口、证据和调度结果。

### 6.3 资产扩展与回流流程

1. 项目执行过程中发现新的域名、IP、入口或服务。
2. 新对象先进入候选资产，携带来源、线索强度和范围判定状态。
3. 系统或研究员完成目标关联与范围判定。
4. 若确认纳入项目，则创建新的前置阶段任务。
5. 项目主阶段保持当前关键路径不变，但以回流提示卡显示“前置阶段任务已补充入队”。

### 6.4 单项审批流程

1. 待验证项进入审批前排队。
2. 审批中心按待处理、风险等级和阻塞影响排序展示。
3. 研究员打开审批详情查看建议理由、具体工具、参数摘要、预期影响、执行前提和停止条件。
4. 研究员做出批准、拒绝或延后决策。
5. 决策结果回写项目详情与任务调度状态。

### 6.5 受控执行流程

1. 审批通过后，任务从 `waiting_approval` 转入可调度状态。
2. 调度模块根据依赖、优先级、并发与速率限制决定是否执行。
3. 任务运行时记录所用 MCP 能力、工具版本、参数摘要和审计信息。
4. 执行完成后产生证据并更新结果。

### 6.6 证据查看与结果确认流程

1. 研究员从项目详情、任务记录或资产详情进入证据详情页。
2. 先查看原始输出和截图，再阅读结构化摘要。
3. 再向下查看关联任务、审批与资产链路。
4. 最终确认结论、置信度和是否需要继续回流补采。

### 6.7 调度干预流程

1. 研究员在项目详情页的任务与调度面板中查看任务状态。
2. 对 `waiting_dependency`、`waiting_approval`、`failed`、`needs_review` 等状态进行人工排查。
3. 可执行暂停、恢复、重试或终止操作。
4. 所有人工干预写入审计链路。

## 7. 关键组件清单

### 7.1 阶段流转组件

用于展示主阶段、当前阶段、已完成阶段、阻塞状态、回流提示和下一步建议。首版需支持“主阶段不重置 + 前置阶段回流提示”的双层表达。

### 7.2 项目摘要组件

用于在项目详情页顶部压缩展示项目名称、目标摘要、主阶段、整体状态、待审批数、风险摘要和最近更新时间。

### 7.3 当前阶段待办组件

用于聚合当前关键任务、等待审批、等待依赖和失败待处理项，并说明阻塞原因与建议操作。

### 7.4 审批卡片与审批详情面板

需要明确展示目标对象、动作类型、风险等级、建议理由、工具能力、参数摘要、预期影响、停止条件和审批决策。

### 7.5 任务队列表格

用于承载任务状态、依赖关系、优先级、调度信息、并发与速率限制、失败原因与人工接管入口。

### 7.6 任务状态标识组件

统一展示以下状态及中文解释：

1. `pending`
2. `ready`
3. `waiting_dependency`
4. `waiting_approval`
5. `scheduled`
6. `running`
7. `succeeded`
8. `failed`
9. `needs_review`
10. `cancelled`

### 7.7 资产画像面板

展示“资产对象当前被识别成了什么”，例如：

1. `22/tcp -> OpenSSH 8.4 版本线索`
2. `3306/tcp -> MySQL 5.7 版本线索`
3. `443/tcp -> nginx + Next.js 前端入口`

### 7.8 资产关系视图

展示 company、domain、subdomain、ip、host、service、website、api、page_entry 等对象之间的关联链路、归属状态和来源说明。

### 7.9 证据详情面板

承载原始输出、截图、结构化摘要、关联任务、关联审批、关联资产、时间线、置信度和最终结论。

### 7.10 MCP 工具配置表

展示 MCP 工具名称、能力类别、版本、风险级别、启用状态和默认限制，并体现“按能力选工具”的平台心智。

## 10. 结论

本次前端原型应以“流程指挥台型”方案为主，构建一套适合单人研究员的后台工作台，而不是普通数据看板或聊天产品。其核心价值在于：

1. 研究员能看懂全局
2. 研究员能管理项目
3. 研究员能追踪阶段推进与阻塞
4. 研究员能集中处理高风险动作审批
5. 研究员能查看资产画像与证据链路
6. 研究员能理解调度状态并人工接管
7. 研究员能在系统设置中感知并管理 MCP 能力与平台控制策略

该文档可直接作为下一步实现计划和高保真原型开发的输入基础。
